# SERA — beauty-saas

Мультитенантный SaaS для салонов красоты. SERA = AI-администратор + инструменты управления (B2B). НЕ «CRM с AI».
Prod: `https://beauty-saas-vert.vercel.app` · GitHub: `vadik88888-prog/beauty-saas` (master = prod) · Supabase: `severincev-beauty` (EU).
Этот файл — правила для исполнителя (Claude Code). Правила общения с владельцем и формат сессии живут в стартовом промте Opus, не здесь. Статус/прогресс — в хендоверах, не здесь.

## 1. КОНСТИТУЦИЯ — нарушение любого пункта = архитектурный баг

1. **AI понимает язык — код принимает решения.** Модель НИКОГДА не создаёт / не переносит / не отменяет запись, не вызывает `book_appointment`, не выбирает скидку, не считает цену, не принимает бизнес-решений. Модель извлекает намерение и параметры — всё остальное делает код.
2. **Запись создаёт только Workflow Engine** — после preview-карточки и явного подтверждения клиента. **Цену считает только Pricing Engine** (`resolveOfferPrice`).
3. **Под `engine=new` модель не получает инструмент записи.** Если модель «не нашла функцию записи» — это НОРМА, а не сбой: по этой причине НЕ звать человека и НЕ сообщать о техническом сбое.
4. **Цена предпросмотра = цена записи.** Карточка и `appointments.price` считаются одним путём; акцию и оффер определяет код, не аргументы модели. Числа обязаны совпадать.
5. **Перехват — по состоянию движка/анкеты, не по тексту модели.** Preview, готовность к записи, валидация триггерятся состоянием Workflow — никогда парсингом фраз («Записываю», «Верно?»). Детектор фразы ломается при перефразировке.
6. **FACT не понижается.** Параметр (услуга, мастер, дата, слот), единожды подтверждённый клиентом, остаётся FACT на все следующие ходы. FACT — накопленное состояние, не снапшот текущего сообщения.
7. **Бренд платформы — только «SERA».** «Алина»/«Alina» и legacy-имена запрещены везде: код, UI, тексты, имена файлов.
8. **Legacy (салон 1) и TMA не трогать** без отдельной явной задачи.

## 2. РАЗВЕДКА И ИСТОЧНИКИ ИСТИНЫ

Иерархия истины: **код > Supabase > Vercel logs > отчёт Claude Code > память модели.** При конфликте верить уровню выше.

- **Твой отчёт — не факт.** Факт — это код, строка в Supabase, запись в Vercel logs.
- Тест владельца говорит одно, твой отчёт другое → **проверять в базе/логах**, не выбирать чью-то версию.
- **Разведка отдельно от правки.** Промт «только читай и доложи» = ни строки кода, только факты. Разведка даёт основание для следующего промта с правкой. Не делать выводов по отчёту без проверки данных.
- `console.log` в Vercel **не виден**. Для проверки «что видит код» — писать в `metadata`/таблицу или читать уровни `warn`/`error`.

## 3. АРХИТЕКТУРА

- Два слоя: **TMA** (клиент, Telegram Mini App) + **Admin** (владелец салона).
- **Booking flow (`engine=new`):** CLIENT → ROUTER → SHADOW FORM (поля = FACT/ASSUMPTION) → READY_TO_BOOK (все 4 поля FACT) → PREVIEW-карточка → CONFIRMATION (`detectConfirmation`, код) → INSERT (Workflow Engine).
- Флаг `booking_engine` (`legacy|new`) на уровне салона, дефолт `legacy`. AI-движок живёт **только** в `lib/ai/administrator/`. Файлы `runAI.ts`, `tools.ts`, `system-prompt.ts` удалены как мёртвый код — не воссоздавать.
- Поле `source` (записи/сообщения): `admin|ai|tma`. Метрика «через SERA» считает только `ai`.
- **Классификация и роутинг — только самая дешёвая модель** (`gpt-4o-mini`). Дорогую модель для роутинга — никогда.
- Имя ассистента задаёт владелец (`admin_name` в `tenant_ai_settings`). Наружу (TMA, клиент) — имя ассистента + орб, без слова «SERA». Внутрь (Admin, владелец) — бренд «SERA». Заглушка до загрузки — `'SERA'`, единый источник — `TmaContext`.

## 4. ПРОД-ПРАВИЛА — это ломает прод

- **Доступа к prod-БД нет.** Изменение схемы → файл `NNN_имя.sql` в `supabase/migrations/`, применяет владелец вручную. В конце задачи КРУПНО вывести SQL с пометкой «⚠️ ПРИМЕНИТЬ В SUPABASE SQL EDITOR». Всегда `ADD COLUMN IF NOT EXISTS`.
- **После `ADD COLUMN` на `tenant_ai_settings`** — в SQL Editor выполнить `NOTIFY pgrst, 'reload schema';`. `loadTenantConfig` делает один SELECT ~10 колонок с `.single()`; если хоть одной колонки нет в кэше PostgREST, весь запрос падает, ошибка глотается молча, и весь конфиг (движок, модель, имя) уходит в дефолт `legacy` без единой строки в логах. *(Тех-долг: `loadTenantConfig` должен логировать `settingsRes.error`, а не глотать его.)*
- **Локальный build ≠ деплой.** `build` зелёный → `git push` → подождать 1–2 мин → Ctrl+Shift+R → ПОТОМ тестировать.
- **Мультитенантность:** `tenant_id` только через `getStaffContext()`, НИКОГДА из тела запроса.
- **Загрузка диалога по `conversationId`** на сервере проверяет `client_id + tenant_id`. Клиентским данным (localStorage, тело) нельзя доверять проверку владельца — чужой UUID игнорируется, диалог находится/создаётся правильный.
- **Серверные компоненты** (без `'use client'`) не могут содержать `onClick` и т.п. → рантайм 500. Hover → CSS-класс. Интерактив → отдельный клиентский компонент.
- **История для OpenAI:** любая роль кроме `user`/`assistant` ремапится в `assistant` (роль `admin` как есть → 400). Фильтр в `conversation-store.ts:loadHistoryWithCount`.
- **Время в UTC, показ по поясу салона.** Форматировать только через `salonTime(iso, tz)` / `getToday()` с явным `tz` из `tenants.timezone`. Сырые `getHours()`, `toLocaleTimeString` без `timeZone` — запрещены (сервер Vercel в UTC даёт сдвиг).
- **Цвет — только из `src/styles/tokens.css`**, без inline hex/rgba. Текст — `--text-muted` (НЕ `--muted` — его перебивает globals.css). `--primary` = `--sage-deep`.
- **Цена со скидкой пишется в `appointments.price`** — её читают все метрики выручки. Экраны после записи берут цену из `appointments.price`, никогда из `services.price`. Скидки не суммируются — применяется выгоднейшая. Промоакции и `client_offers` — раздельные механизмы; в одной записи заполнен один источник скидки.
- **Телефоны — только цифры** (убирать `+`, пробелы, скобки, дефисы) при поиске и записи. Дубликат клиента → HTTP 409 с данными существующего. TG username в БД — без «@» (`@` только в рендере).
- **Удаление услуги с историей** блокирует FK (`appointments.service_id`) → только `is_active=false`. `is_active` (учитывается в метриках) ≠ `show_in_storefront` (видна клиентам).

## 5. КАК РАБОТАЕМ

- **Vertical slice:** одна задача за раз (миграция → API → фронт → проверка деплоя). `git commit` перед каждой задачей. Один промт — одна правка, дальше проверка на проде.
- **Контекст ищешь сам** (код, CLAUDE.md, скиллы). НЕ проси у владельца пути, номера строк, куски кода — находи по имени функции/символа.
- **Отчёт — одна строка на изменение:** что и зачем. Без пересказов и больших кусков кода.
- После зелёного `npm run build` сам коммитишь и пушишь, **останавливаешься** — владелец проверяет на проде. Критично не хватает данных или есть противоречие с кодом → задай **один** вопрос, не угадывай.
- `.claude/` не трогать. Дизайн — точка правды `docs/DESIGN_SYSTEM_REFERENCE.html`, страницы собираем скиллом `/sera-page`. `SeraOrb` не трогать без отдельной задачи.
- Планы — `docs/` (`PLANS_INDEX.md`). Журнал сделанного — `docs/HISTORY.md`.
