# SERA — beauty-saas

Мультитенантный SaaS для салонов красоты. SERA = бренд платформы + AI-администратор (B2B-продукт).
Prod: `https://beauty-saas-vert.vercel.app` · GitHub: `vadik88888-prog/beauty-saas` (master = prod) · Supabase: `severincev-beauty`, EU

---

## Стек и команды

- Next.js 16 (App Router, Turbopack) + TypeScript + Tailwind v4 + shadcn/ui
- Supabase (PostgreSQL + RLS) · framer-motion 12 · Grammy.js (bot) · OpenAI GPT-4o-mini (`OPENAI_API_KEY`)
- Route groups: `(tma)` `(admin)` `(auth)` `(onboarding)` · Middleware: `proxy.ts` (НЕ `middleware.ts`)
- Сборка: `npm run build` · Смоук: `bash scripts/smoke-test.sh` · Деплой: `git push` (Vercel автодеплой)

## Железные правила (не нарушать)

1. **Бренд платформы — только «SERA».** Слова «Алина»/«Alina» запрещены везде: код, UI, тексты, имена файлов.
2. **Имя ассистента** задаёт владелец (`admin_name` в `tenant_ai_settings`). В TMA наружу показываем ИМЯ АССИСТЕНТА из базы + орб, не слово «SERA». Заглушка до загрузки — `'SERA'`. Единый источник — `TmaContext`.
3. **Два контекста имени:** наружу (клиент, TMA) — имя ассистента, без слова SERA. Внутрь (владелец, Admin) — бренд SERA везде.
4. **Мультитенантность:** каждый Admin API-запрос — только через `getStaffContext()`. `tenant_id` НИКОГДА из тела запроса (иначе клиент видит чужой салон).
5. **Цвет — единственный источник `src/styles/tokens.css`.** Никаких hex/rgba inline. Для текста — `--text-muted` (`--muted` перекрывается globals.css светлым shadcn-значением).
6. **`--primary` = зелёный (`--sage-deep`).** Чёрная кнопка (`--ink`) — вторичный вариант, не primary. `.sera-btn--primary` = `--ink`, `.sera-btn--sera` = `--sage-deep`; ни одна не читает `--primary`.
7. **Поле `source` у записей/сообщений:** `admin` / `ai` / `tma`. Метрика «через SERA» считает только `ai`. При новых ролях/значениях — проверять CHECK-констрейнты в БД.
8. **Миграции БД:** изменение схемы → отдельный файл `NNN_имя.sql` в `supabase/migrations/` (NNN — следующий порядковый номер). Применяет владелец вручную. Claude Code создаёт файл и в конце задачи КРУПНО выводит SQL с пометкой ⚠️ ПРИМЕНИТЬ В SUPABASE SQL EDITOR. Всегда использовать `ADD COLUMN IF NOT EXISTS`. Доступа к prod-базе нет.
9. **Деплой:** после `build зелёный` → `git push` → подождать 1–2 мин → Ctrl+Shift+R → ПОТОМ тестировать. Локальный build ≠ прод.
10. **Темп:** маленькие шаги, одна задача. Перед правкой — диагностика. `git commit` перед каждой задачей.
11. **Серверные компоненты (без `'use client'`) не могут содержать обработчики событий** (`onClick`, `onMouseOver` и т.п.) — Next.js App Router упадёт в рантайме с 500. Hover → CSS-класс в `<style>`. Интерактив → выделить в отдельный клиентский компонент. Не делать всю страницу клиентской ради мелкого эффекта.
12. **Telegram username в БД — без «@».** При ручном вводе срезать ведущий «@» (`replace(/^@+/, '')`). При показе добавлять «@» только в рендере, не в данных. Двойной `@@` — это баг ввода, не данных.
13. **Персональные скидки** хранятся в `client_offers`. Итоговая цена со скидкой пишется в `appointments.price` — именно это поле читают все метрики выручки. Поле `source ('salon'|'sera')` — задел: AI пишет туда как `'sera'` позже. Скидки не суммируются — применяется выгоднейшая.
14. **Цена записи считается только на сервере** единым калькулятором. Фронт цену не присылает. Расчёт не дублировать в роутах — только через `resolveOfferPrice`.
15. **Цена на экранах после записи** (Success, карточки записей) берётся только из `appointments.price`. Никогда из `services.price`.
16. **Скидки не суммируются** — применяется выгоднейшая. Персональные офферы и промоакции — раздельные механизмы; в одной записи заполнен только один источник скидки.
17. **Загрузка фото:** bucket `service-images` — для услуг, `master-photos` — для мастеров. Эндпоинт `/api/admin/upload` принимает `?bucket=` (дефолт `master-photos`). `image_url` хранит полный публичный HTTPS-URL из Supabase Storage; `next.config.ts` разрешает `**.supabase.co` и `**.supabase.in`.
18. **Удаление услуги с историей запрещено:** `appointments.service_id` — FK без `ON DELETE`, удаление физически блокируется базой. Если у услуги есть записи → только `is_active = false` (деактивация). Проверка через `GET /api/admin/services/[id]` → `canDelete`.
19. **Числовые поля форм:** без стрелок-спиннеров (CSS `.no-spinner`), без мешающего нуля — новая запись открывается с пустым полем и placeholder. `inputMode="decimal"` / `"numeric"` для цифровой клавиатуры на телефоне. Обязательные поля (цена, длительность) валидируются до отправки в API — при пустом/некорректном значении показывать `formError`, не пускать запрос.
20. **Тумблеры услуги различны:** `is_active` — услуга существует и учитывается в метриках; `show_in_storefront` — видна клиентам в публичной витрине. Услуга может быть `is_active=true` и `show_in_storefront=false` одновременно. При деактивации через «удаление с историей» — только `is_active=false`, `show_in_storefront` не трогать.
21. **История для OpenAI:** перед отправкой любая роль кроме `user` и `assistant` приводится к `assistant`. Роль `admin` (ответ сотрудника в чат-панели) никогда не уходит в OpenAI как есть — иначе 400 Invalid messages. Фильтр и ремап живут в `conversation-store.ts:loadHistoryWithCount`.
22. **AI-движок — только `lib/ai/administrator/`.** Файлы `lib/ai/runAI.ts`, `lib/ai/tools.ts`, `lib/ai/system-prompt.ts` удалены как мёртвый код. Не воссоздавать; новые компоненты AI добавлять только внутрь `administrator/`.
23. **Проактивные сообщения (напоминание, опрос после визита, возврат клиентов через `trigger-client-message`) сейчас НЕ пишутся в таблицу `messages` — AI их не видит.** При работе над памятью клиента или персонализацией нужно дублировать их в `messages` с ролью `assistant` и добавлять это значение в whitelist-фильтр `conversation-store.ts`.
24. **Загрузка диалога по `conversationId` на сервере обязана проверять `client_id + tenant_id`.** Клиентским данным (localStorage, тело запроса) нельзя доверять проверку владельца — если диалог не принадлежит текущему клиенту, сервер игнорирует UUID и создаёт/находит правильный диалог. Эталон — `ConversationStore.load()` и маршрут `/api/ai/chat/history`.
25. **Классификация и маршрутизация сообщений — только самая дешёвая модель** (`gpt-4o-mini` или аналог). Дорогую модель для роутинга не использовать никогда.
26. **AI понимает язык — код управляет бизнесом.** Модель никогда не создаёт запись, никогда не вызывает `book_appointment` напрямую, никогда не принимает бизнес-решения и никогда не определяет цену. Создание записи — только через Workflow Engine. Расчёт цены — только через Pricing Engine (`resolveOfferPrice`). AI извлекает намерение и параметры; всё остальное — код.
29. **Время записей хранится в UTC, показывается по поясу салона.** В серверных компонентах Admin-панели любое время из БД форматировать только через `salonTime(iso, tz)` из `@/lib/utils/date` с явным `tz` из `tenants.timezone`. Сырые `d.getHours()`, `toLocaleTimeString` без `timeZone`, `formatTime` без tz — запрещены: сервер Vercel работает в UTC и даёт сдвиг на часы.
27. **Перехват — по состоянию движка, не по тексту модели.** Любая серверная логика (preview записи, готовность к `book_appointment`, валидация) триггерится состоянием Workflow/анкеты — никогда парсингом фраз модели («Записываю», «Верно?»). Детектор фразы ломается при перефразировке. Состояние движка — нет.
28. **FACT не понижается.** Параметр записи (услуга, мастер, дата, слот), единожды подтверждённый клиентом в диалоге, остаётся FACT на все последующие ходы. Пересчитывать `source` только по последнему сообщению нельзя — FACT — накопленное состояние, не снапшот текущего хода. Касается теневой анкеты и любой будущей формы.

## Контракт качества (соблюдать в каждой задаче)

- **Ноль фейка:** только реальные данные из API/БД. Нет данных → пустое состояние, не выдумка.
- **Данные сначала:** новая фича строится на реальной таблице и механике. AI, дополнительная логика и умные подборки подключаются к готовым данным позже — не наоборот. Если данных для расчёта нет, показываем пустое состояние, не придумываем цифры.
- **Живые кнопки:** каждая кнопка/ссылка ведёт в реальное место. Мёртвых `onClick` / `href="#"` нет.
- **Телефон (~375px)** обязан выглядеть идеально. Тяжёлые desktop-сетки на узком экране упрощать.
- **Цвет — акцентами** (полосы, точки, бейджи). Большие площади не заливать брендовым тинтом.

## Постановка задач (как работаем)

- Контекст ищешь сам: у тебя код, CLAUDE.md, скиллы. **НЕ проси** у владельца пути, номера строк, куски кода — находи по имени функции/символа.
- **Разведка отдельно от правки.** Если задача начинается «только читай и доложи» — только читать и докладывать фактами, код не трогать. Разведка даёт факты для следующего промта с правкой.
- Делаешь сразу и показываешь результат. Отдельный «план на согласование» не нужен, если не попросили.
- Отчёт — **одна строка на изменение**: что и зачем. Без пересказов и без вставки больших кусков кода.
- После зелёного `npm run build` сам коммитишь и пушишь (скилл `deploy`), останавливаешься — владелец проверяет на проде.
- Промт = решение + границы + критерий готовности. Если критичного не хватает или есть противоречие с кодом — задай **один** вопрос, не угадывай.
